## The problem

A migration that passes every test on a dev database with twelve rows in it can
still take production down. `SET NOT NULL` on a column that is 3% null fails on
contact. A cascade delete on a foreign key with a long fan-out tail is an outage,
not a cleanup. The shape of the data decides the answer, and the shape is exactly
what a dev database does not have.

<CardGrid>
	<Card title="Shape, not rows">
		A fixture is statistics — row counts, null fractions, cardinality, fan-out
		distributions. No rows. It commits to your repo and is small enough to review.
	</Card>
	<Card title="One verdict, two readers">
		The CLI, the MCP server, and the GitHub Action all render the same verdict

	<Card title="Confidence is part of the answer">
		A verdict is capped by the weakest fact it rests on. If rowshape could only

	<Card title="Built for the agent loop">

	</Card>
